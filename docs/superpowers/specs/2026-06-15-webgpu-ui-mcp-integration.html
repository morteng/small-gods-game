WebGPU-native UI
+ MCP into the running game

Two pivots, one spine. Replace the DOM game/dev UI with a WebGPU-native, immediate-mode pixel-art interface — and drop iframe embedding for an MCP server that lets Claude co-inhabit the live world. Both are thin front-ends onto a single command/query bus.

★North star

The world is the interface.

A god watches; it does not read dashboards. The screen is almost all world — terrain, settlements, mortals living — with chrome that appears on demand and dissolves when done (the summoned Time bar is the template; extend that everywhere). Information surfaces when you focus, never permanently. Pixel-art aesthetic, world palette, modern and restrained, for a mature audience that doesn't need hand-holding.

And the same world is co-inhabited by Claude — as Fate, divine co-pilot, dev surface, and world author. The human's clicks and Claude's tool-calls are the same kind of event: intents on one bus, recorded to the canonical log, deterministic and replayable.

Design lineage: RimWorld & Dwarf Fortress (inspect-on-demand, no permanent stat bars) · Hyper Light Drifter / Hades (diegetic minimalism) · Songs of Conquest / Norland (mature pixel art) · Caves of Qud (deep info, surfaced contextually). The shared thesis: information on demand, not on display.

Because every MCP-driven mutation is recorded as an authored event exactly like a player action, snapshot/replay reconstructs Claude's hand too. The UI's action surface and the MCP tool surface are thin adapters — "who asked" is decoupled from "what happens." Wins: determinism, Node-testable replay, one place to enforce power/cost/permission.

Wire protocol: { id, type:'req'|'res'|'event', method, params, result, error }. req/res are correlated commands & queries; the event push channel streams the world's pulse (belief/mood crossings, deaths, prayers) — that's what lets Claude be Fate. Queries read the current snapshot; commands enqueue and apply at the next tick boundary, then return deltas.

DGenerated UI skin — research-grounded

The chrome is painted by the image-gen pipeline, re-skinnable by era. Claude + engine own structure; the model owns only the cosmetic skin. Deep research: wf_19624f8e-9b7 · 18 sources · 73 claims · 23/25 verified.

The one verified principle: never trust raw diffusion output. Latent VAEs compress to ~⅛ resolution and discard high-frequency detail — so output has off-grid "pixels", palette drift, distorted text, and unreliable alpha. Four properties are enforced in code, never trusted from the model: pixel grid, palette, alpha, text.

The strategic fork

Lane A · closed-API (start here)

Existing pipeline: FLUX.2 Klein 4B multi-reference img2img + single-seed + chroma-key + gate + deterministic quantize. Great for hard-edged frames, buttons, icons. Lowest new infra; approximates coherence via style-anchor + locked palette.

Lane B · self-hosted (defer)

Stand up an SDXL/FLUX-Dev lane to unlock LayerDiffuse (true soft-edged alpha) + IP-Adapter/InstantStyle (tightest kit coherence). Worth it only if Lane A hits a wall on soft glow/ornament alpha or coherence — decide empirically.

Model: FLUX.2 Klein 4B (Apache-2.0, ~13GB VRAM, sub-second, unified t2i + multi-ref) is the verified default — but its card makes no alpha and no pixel-grid guarantees (matching the in-house gemini-3 opaque-checkerboard finding), so both are gated empirically, never assumed. The architecture is durable; the specific model is not.

Open questions → S3.5 spec

Spike first (decides the fork): does gemini-2.5-flash / FLUX.2 Klein via OpenRouter yield usable keyable alpha for UI ornament — or does the opaque-checkerboard mode recur, forcing Lane B? Plus: per-asset cost/latency for a full early-medieval kit; make quantize byte-deterministic (k_seed); do tileable 9-slice tiles survive quantization without seams (seam-aware quantize)?

✓Resolved decisions

All seven open questions resolved — these are constraints for the spec, not options.

Q1 · FontsOne family, two atlases

A single high-legibility pixel typeface drives both — crisp bitmap blit for integer-scaled HUD, MSDF from the same TTF for zoom-scaling world labels. Primary: Pixel Operator (OFL, proportional, bold). Alt for a terminal/tech register: Departure Mono. Swappable — it's just an atlas.

Why: HUD and world labels share one identity; bitmap stays pixel-crisp, MSDF stays sharp under zoom.

Q2 · Packagingmcp/ dir · stdio to Claude · embedded WS

New top-level mcp/ run via tsx. Transport to Claude = stdio, registered in .mcp.json (Claude Code spawns it, like Playwright). That same process runs the WebSocket server the browser connects to (env port, default 7777). Vite does not own it. Standalone npm run mcp for other clients.

Why: stdio is the standard, lowest-friction MCP transport and lets Claude Code manage lifecycle; the WS server rides along in-process.

Q3 · PowerDiegetic orb, no resident number

The orb encodes power by fill + pulse (pulse = can afford the selected action). Hold/hover peeks the exact value + regen rate; never permanently on screen. Validated in S3.

Why: matches stat-minimalism while keeping the number one gesture away.

Q4 · Human ↔ ClaudeFree-for-all + visible attribution, no locks

Commands serialise through the one bus (order = tick queue) — no data race, so locks are premature. Every command carries author: 'player' | 'fate' | 'claude', surfaced in the event log and as a transient in-world marker ("Fate stirs…" + the entity flashes).

Why: serialisation already removes the race; the real risk is legibility — solved by attribution, not concurrency control.

Q5 · Legacy UIStudio + dev panels stay on the DOM island — not ported

Canvas UI is game-facing only. Studio & heavy dev panels are tools, not player UX; porting is low-value churn. The MCP dev/debug surface (S5) absorbs most dev-panel need — Claude spawns/inspects/time-travels/screenshots via tools — so panels retire by replacement, not rewrite.

Why: spend the canvas budget on player UX; let MCP tools eat the dev panels.

Q6 · WS auth127.0.0.1 + auto-generated shared token, no pairing

MCP server mints a token at boot; the dev server hands it to the browser (file or /mcp-token dev endpoint); the browser sends it in the WS handshake. Origin-checked. No interactive pairing on localhost. Revisit only if exposed beyond localhost.

Why: localhost + shared secret is sufficient for a same-machine dev/play setup.

Q7 · Mobile WebGPUNamed minimum targets, hard gate otherwise

Test targets: desktop Chrome/Edge/Firefox + Safari 18+; mobile iOS Safari 18+ and Chrome Android 121+. S2 gestures tested on those. No WebGPU → a clear "WebGPU required" screen, no fallback.

Why: per the WebGPU-hard-requirement directive; naming real devices means touch parity is tested where it ships.

✕Non-goals (for now)

  • Canvas2D / WebGL UI fallback — WebGPU is required.
  • Re-implementing IME / text-shaping on canvas — the DOM island owns text.
  • Iframe embedding / postMessage host API — replaced by MCP; removed in S7.
  • Resurfacing the full stat panel — minimalism first; add back only what play proves.
